**Operational lessons (hard-won — fold into procedure):**
- **HA → HomeSynapse handoff: POWER-CYCLE THE DONGLE FIRST.** An abrupt HA teardown leaves STALE NCP state → first launch fails `zigbee.auto_detect_failed bytesReceived=0` (looks like a transport defect; it is NOT). A dongle power-cycle clears it and formation succeeds. (pyserial/universal-silabs-flasher probing fine is the tell the radio is alive.)
- **Only ONE process may own `/dev/zigbee`.** Stop HA / the prior app before launching; a double-launch throws `BindException` on :7070.
- **The systemd unit is NOT bench-usable** (`PrivateDevices=yes` blocks `/dev`). Run the `installDist` launcher directly as a `dialout` user (nohup + tee).
- **Reliable stop:** `pkill -f "[c]om.homesynapse.app.Main"` (bracket avoids self-match). `pkill -f homesynapse-app` self-kills the launch command.
- **Permit-join window = 254 s max; each launch re-opens it** — reset the device immediately after launch. A `zigbee.reopen_no_target` means the dongle got bumped/re-enumerated.
- **Un-wired config keys:** `permit_join_duration` NOW wired (M9.4-PJ). `channel` wiring rides `M9.4-TCJ §B` (pin ch20 = the ZHA-proven channel; the bench formed on ch25 fallback). `adapter_type` still un-wired (auto-detect only) — schema'd + error-suggested but no adapter read; low priority (auto-detect works).
- **Custody is PRESERVED** across the fix: the ch25 network + keys persist in `~/hs-bench/data/zigbee/`, so the app RESUMES after M9.4-TCJ lands — no re-formation needed (unless §B pins a new channel, which re-forms once).
- **The bearer token rotates: re-`cat` `$TOKEN` after ANY app relaunch, in EVERY fresh ssh session** (`TOKEN=$(cat ~/hs-bench/config/initial_api_token)`). A token cat'd before the current launch 403s (`invalid, expired, or revoked`) — it looks like an API defect and is not. Paid for at iteration 5b (2026-07-08).
- **After `pkill`, sleep ~5 s before relaunching.** An instant relaunch races the dying instance for `/dev/zigbee` (ONE process owns the port); the old instance's `transport_failed`/`reopen_no_target` during its own shutdown is death-rattle noise, not a defect. Paid for at iteration 5b (boot 2 consumed by the race).
- **Use `tools/bench.sh` for ALL bench operations (adopted 2026-07-10, the acceptance run):** `bench.sh {start|stop|restart|status|health|log|entities|runs|events|state <ulid>}`. `start` launches AND polls to a DECISIVE verdict (RADIO UP / INTEGRATION FAILED with evidence) — no `$LOG` juggling, no sleep/grep races, no double launches (it refuses if one is running). Lives at `~/bench.sh` on the Pi; source of record in this repo.
- **ModemManager is DISABLED on the bench Pi** (`systemctl disable --now ModemManager`, 2026-07-10) — it claims freshly-enumerated serial ports and is a confounder for every port incident. Re-disable after any OS reinstall.
- **The Hue LCA017 factory reset that actually works: the Hue app over Bluetooth** (scan the bulb's QR → pair BT → factory reset). The 6× power-cycle dance failed 3+ consecutive attempts on this firmware — do not budget bench time for it.
- **Clearing a foreign/stale network off the NCP:** patch the bellows CLI's v13 incompatibility (`sed` the `v[0] == t.EmberStatus.NOT_JOINED` check in `bellows/cli/network.py` to tolerate scalar `sl_Status`), run `bellows -d /dev/zigbee -b 115200 leave`, then **`pkill -f bellows`** (the CLI crashes cosmetically AFTER sending the leave and hangs holding the port). The next app boot takes resume()'s restore branch (`zigbee.network_restored_from_parameters`) and re-forms the stored network from custody — silicon-proven 2026-07-10. zigpy-cli 1.2.1 has NO network-leave command; don't reach for it.
- **NO mid-run `sudo reboot` unless ruled** — the remedy ladder is pkill → sleep → dongle power-cycle. A reboot resurrects anything autostart-enabled and voids the incident timeline (paid for 2026-07-09/10: the foreign-network writer window is un-witnessed because of it).
- **A pulled-and-re-seated dongle does NOT auto-recover (known defect, R-row 2026-07-10):** the watchdog detects and retries honestly but `reopen_no_target` persists after re-enumeration (new USB devnum) — restart the app (`bench.sh restart`) to recover. Fix is a queued post-soak micro-WU.
